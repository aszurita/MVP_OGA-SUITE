Los 4 Contenedores Docker

Qué va dentro de cada imagen, por qué y qué expone

⚡
Contenedor 1 — Nginx Reverse Proxy
Imagen base: nginx:alpine · Puerto expuesto: 80 (local) / 443 en Azure

¿Qué hace?

  • Recibe todo el tráfico HTTP/HTTPS de usuarios
  • Ruta / → sirve el React app (estático)
  • Ruta /api/ → proxy hacia API container :8510
  • Maneja CORS, headers de seguridad, gzip
  • Sirve archivos estáticos con caché agresiva (img, css, js)

¿Por qué existe?

  • Un solo punto de entrada al sistema
  • El banco solo abre UN puerto en el firewall
  • Puede escalar React y API independientemente
  • SSL termination en un solo lugar
⚛️
Contenedor 2 — React App (Frontend)
Build multi-stage: node:18-alpine → nginx:alpine · Solo sirve archivos dist/

Proceso de build (multi-stage)

  • Stage 1 (builder): node:18-alpine → npm ci → npm run build
  • Stage 2 (runner): nginx:alpine → copia solo el dist/
  • Imagen final: ~25 MB (sin node_modules)
  • Variables de entorno inyectadas en build time con Vite

¿Qué incluye?

  • ✅ src/ (React source compilado)
  • ✅ Assets estáticos (img, font, css)
  • ✅ dist/index.html + bundles JS
  • ❌ node_modules (excluido)
  • ❌ Archivos fuente .jsx (excluido)
  • ❌ js/ legacy de 399 archivos (excluido en prod)
🔌
Contenedor 3 — API Backend
Imagen: node:18-alpine · Puerto interno: 8510

¿Qué contiene?

  • El servidor Node/Express que corre en gobinfoana01-2:8510
  • Endpoints: /query, /insert, /update
  • Conexión a SQL Server (PROCESOS_BI)
  • Capa de Redis para cachear respuestas
⚠️ Nota: Si el código fuente de la API está en el servidor del banco, necesitas obtenerlo para containerizarlo. Si solo tienes el binario/URL, lo referencias como servicio externo en el docker-compose.yml y solo containerizas el frontend + Redis.
⚡
Contenedor 4 — Redis Cache
Imagen oficial: redis:7-alpine · Puerto interno: 6379

¿Para qué lo usas?

  • Cache de todos los términos del glosario (TTL 5 min)
  • Cache de dominios y mapas de dominios
  • Términos recientes por usuario (ya los tienes en state)
  • Rate limiting en la API
  • Sesiones de usuario (si implementas auth)

Config mínima

  • Imagen: redis:7-alpine (10 MB)
  • Puerto: 6379 (solo interno, nunca expuesto)
  • Persistencia: appendonly yes
  • Memoria máxima: 256mb
  • En Azure: reemplazar por Azure Cache for Redis

¿Para Qué Específicamente Redis?

Casos de uso concretos en OGA-SUITE

📚
Cache de Términos
El endpoint getTerminosAll() hace una query pesada a SQL. Redis la cachea por 5 minutos. Si hay 50 usuarios simultáneos, SQL recibe 1 query en vez de 50.
🕐
Términos Recientes
Hoy los guardas en localStorage. Con Redis puedes sincronizarlos entre dispositivos del mismo usuario usando su ID bancario como key.
🗺️
Cache de Dominios
t_mapa_dominios casi no cambia. Redis lo mantiene en memoria y sirve instantáneo. TTL de 30 minutos.
🔍
Resultados de Búsqueda
Búsquedas frecuentes como "cliente", "transacción" se cachean como search:cliente → [términos] con TTL de 2 minutos.
🚦
Rate Limiting
Limitar requests por IP o usuario. Ej: máx 100 requests/minuto. Fundamental en entornos bancarios para evitar abusos.
🔐
Sesiones (futuro)
Cuando agregues autenticación bancaria (LDAP/AD), las sesiones van en Redis. Permiten logout centralizado y expiración automática.
Ejemplo de integración Redis en la API (Node.js) apiService.js — con cache
// Patrón Cache-Aside: primero busca en Redis, si no hay va a SQL async function getTerminosAll() { const cacheKey = 'terminos:all'; // 1. Intentar desde cache const cached = await redis.get(cacheKey); if (cached) return JSON.parse(cached); // 2. Cache miss → consulta SQL Server const result = await query(`SELECT * FROM procesos_bi.dbo.T_terminos WHERE activo = 1`); // 3. Guardar en cache por 5 minutos await redis.setex(cacheKey, 300, JSON.stringify(result)); return result; }

docker/frontend/Dockerfile Multi-stage build · Imagen final ~25MB
# ── STAGE 1: Build de React con Vite ───────────────────────────────── FROM node:18-alpine AS builder WORKDIR /app # Copiar solo package files primero (aprovechar cache de Docker) COPY MOCKUP-GLOSARIO/Desarrollo/package*.json ./ # Instalar dependencias (solo producción para el build) RUN npm ci # Copiar el código fuente React (el .dockerignore excluye js/ legacy) COPY MOCKUP-GLOSARIO/Desarrollo/ . # Variables de entorno para el build ARG VITE_API_URL ENV VITE_API_URL=$VITE_API_URL # Compilar la aplicación RUN npm run build # ── STAGE 2: Servir con Nginx (imagen mínima) ───────────────────────── FROM nginx:alpine # Copiar solo el resultado del build (dist/) COPY --from=builder /app/dist /usr/share/nginx/html # Configuración de Nginx para SPA (React Router) COPY docker/nginx/nginx.conf /etc/nginx/conf.d/default.conf EXPOSE 80 CMD ["nginx", "-g", "daemon off;"]
docker/nginx/nginx.conf Reverse proxy + SPA fallback
server { listen 80; server_name _; # Gzip para mejor rendimiento gzip on; gzip_types text/html text/css application/javascript application/json; # Servir React SPA location / { root /usr/share/nginx/html; index index.html; # Crucial para React Router — rutas como /glosario, /politicas try_files $uri $uri/ /index.html; } # Proxy hacia la API (evita CORS en el browser) location /api/ { proxy_pass http://api:8510/; proxy_http_version 1.1; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; } # Cache agresiva para assets estáticos location ~* \.(jpg|jpeg|png|gif|ico|css|js|woff2?)$ { expires 30d; add_header Cache-Control "public, immutable"; } }

Despliegue en Azure

Servicios de Azure que usarías — del registro de imágenes hasta la app en producción

📦
Container Registry
Azure Container Registry (ACR)
Donde subes y versionas tus imágenes Docker. Es como Docker Hub pero privado y dentro de Azure. Cada push de código genera una nueva imagen.
🚀
Runtime de Contenedores
Azure Container Apps
Corre tus contenedores sin gestionar VMs. Auto-scaling, HTTPS automático, integración con ACR. Más simple que AKS para este proyecto.
⚡
Redis Managed
Azure Cache for Redis
Redis administrado por Azure. Sin mantenimiento, con alta disponibilidad, backups automáticos. En prod reemplaza al contenedor Redis local.
🌐
Entrada / CDN
Azure Front Door
CDN global + WAF (firewall de aplicaciones) + HTTPS. Los assets estáticos del React (JS, CSS, img) se sirven desde el edge, ultra rápido.
🗄️
Base de Datos
Azure SQL (o keep server)
Puedes migrar PROCESOS_BI a Azure SQL Database (compatible con SQL Server 100%). O dejar gobinfoana01-2 y conectar desde Container Apps via VNet.
🔑
Secretos
Azure Key Vault
Almacena strings de conexión, contraseñas de DB, API keys. Container Apps los consume automáticamente. Nunca secretos en el código.
🔁
CI/CD
Azure DevOps Pipelines
Pipeline automático: push a main → build imagen → push a ACR → deploy a Container Apps. El banco ya tiene Azure DevOps.
📊
Observabilidad
Azure Monitor + App Insights
Logs, métricas, alertas, trazas de requests. Ves exactamente qué query SQL está siendo lenta o cuántos cache hits tiene Redis.

Migración a Podman (Cuando Llegue el Momento)

La gran noticia: no necesitas reescribir nada

🦭 Podman es compatible 100% con Docker

Podman lee los mismos Dockerfiles, el mismo docker-compose.yml (via podman-compose), y usa los mismos comandos. La migración es literalmente instalar Podman y cambiar alias.

# Instalar podman-compose pip install podman-compose # Los mismos comandos — solo cambia "docker" por "podman" podman build -f docker/frontend/Dockerfile -t oga-frontend . podman-compose up --build # O crear alias para no cambiar hábitos alias docker=podman alias docker-compose=podman-compose
✅ Ventajas de Podman para el banco
  • Rootless: corre sin root — más seguro
  • Sin daemon: no hay proceso en background
  • RHEL/UBI: imágenes certificadas para empresas
  • OpenShift compatible: Azure Red Hat OpenShift
  • Sin licencia: Docker Desktop ya cobra en empresas grandes
📋 Qué cambia al migrar
  • NADA en los Dockerfiles
  • NADA en el docker-compose.yml
  • NADA en el código de la app
  • Solo: instalar Podman + podman-compose
  • Opcional: ajustar variables de entorno de la red
